ContextForge — Pluggable Token Storage

Architect Design Document · Feature #5402

Status: Pending Architect Approval  |  Prepared for: Architect Review  |  Environments: Local Dev (Vault + PostgreSQL) · Production (Vault + PostgreSQL HA)

1Executive Summary

Problem. Every OAuth access_token and refresh_token issued through ContextForge's Authorization Code flow is hardwired into a single oauth_tokens database table. Enterprise clients operating HashiCorp Vault cannot redirect credential storage without forking the core service.

Solution. Implement VaultTokenBackend as a new token storage option alongside the existing database implementation. The backend is selected via OAUTH_TOKEN_BACKEND environment variable. The existing database code remains completely untouched — no refactoring, no interface extraction. This is a parallel implementation, not a replacement.

Vault-specific features. (1) Two new client-callable endpoints — GET /vault/authorize/{server_id} and GET /vault/callback — let clients initiate the Vault credential flow using only their virtual server URL. (2) Vault secrets are stored at path team_id/server_id/email where server_id is derived from the upstream MCP URL (from gateways.url). (3) An optional in-memory token cache with configurable TTL mitigates Vault API latency for high-throughput deployments. Note: The existing database OAuth flow is not modified — these features only apply when OAUTH_TOKEN_BACKEND=vault.

3Client Access Model — Why gateway_id Is Never Exposed

A client's MCP configuration references only the virtual server URL:

// Client MCP config (claude_desktop_config.json, Cursor settings, etc.)
{
  "servers": {
    "git-server": {
      "type": "streamable-http",
      "url":  "http://localhost:4444/servers/647ad7b348044bce8fa27a2157b00a0d/mcp",
      "headers": { "Authorization": "Bearer your-token-here" }
    }
  }

The server_id (647ad7b3…) is the virtual server primary key in the servers table. Underlying gateway_id values are internal — clients must not need to know them.

5AbstractTokenBackend Interface

Key Design Decision — gateway_id as the interface discriminator.
All five call sites (oauth_router.py, tool_service.py, gateway_service.py, resource_service.py, admin.py) already hold a gateway_id and pass it to TokenStorageService. The interface therefore keeps gateway_id as its parameter — zero changes to any call site.

The client never supplies gateway_id. The client only knows the virtual server URL (/servers/{server_id}/mcp). The vault_router walks the resolution chain server_id → server_tool_association → tools.gateway_id → gateways.url to find the gateway internally — the client is never involved in this lookup.

Vault path key = gateways.url, not gateway_id. VaultTokenBackend receives gateway_id and immediately calls _resolve_mcp_url(gateway_id) to fetch gateways.url from the DB. That URL becomes the human-readable Vault path segment. DatabaseTokenBackend uses gateway_id directly as its FK — no lookup needed. 
from abc import ABC, abstractmethod
from dataclasses import dataclass
from datetime import datetime

@dataclass
class TokenRecord:
    """Plain dataclass — no SQLAlchemy dependencies."""
    gateway_id: str           # gateways.id (UUID) — used by DB backend
    mcp_url: str              # gateways.url — resolved by VaultTokenBackend; used as Vault path key
    user_id: str              # OAuth provider user ID
    app_user_email: str       # ContextForge user identity
    access_token: str         # Plain-text (each backend handles encryption differently)
    refresh_token: str | None
    token_type: str           # Always "Bearer"
    expires_at: datetime | None
    scopes: list[str]
    created_at: datetime
    updated_at: datetime


class AbstractTokenBackend(ABC):
    """
    Backend-agnostic token storage interface.

    All methods receive gateway_id. Each backend resolves it appropriately:
      DatabaseTokenBackend  → uses gateway_id directly as FK in oauth_tokens table
      VaultTokenBackend     → calls _resolve_mcp_url(gateway_id) → gateways.url → Vault path key

    The CLIENT never passes gateway_id. It only knows server_id (virtual server URL).
    The vault_router resolves: server_id → server_tool_association → tools.gateway_id
    and then calls TokenStorageService with that gateway_id — exactly as existing call sites do.
    """

    # Write — called at OAuth callback after IdP returns tokens
    @abstractmethod
    async def store_tokens(
        self,
        gateway_id: str,        # UUID from gateways.id — passed by all existing call sites
        user_id: str,           # OAuth provider user ID
        app_user_email: str,    # ContextForge user email
        access_token: str,
        refresh_token: str | None,
        expires_in: int | None,
        scopes: list[str],
    ) → TokenRecord: ...

    # Read — called on every tool call / health-check / resource fetch
    @abstractmethod
    async def get_user_token(
        self,
        gateway_id: str,
        app_user_email: str,
        threshold_seconds: int = 300,
    ) → str | None: ...       # Auto-refreshes if within threshold_seconds of expiry

    # Metadata — called by admin status API
    @abstractmethod
    async def get_token_info(
        self, gateway_id: str, app_user_email: str,
    ) → dict | None: ...      # Returns non-sensitive metadata (scopes, expiry, status)

    # Revocation — called at user logout or admin revoke
    @abstractmethod
    async def revoke_user_tokens(
        self, gateway_id: str, app_user_email: str,
    ) → bool: ...             # Returns True if deleted, False if not found

    # Maintenance — called by scheduled cleanup job
    @abstractmethod
    async def cleanup_expired_tokens(
        self, max_age_days: int = 30,
    ) → int: ...
    # DatabaseTokenBackend: deletes expired DB rows.
    # VaultTokenBackend:    returns 0 — Vault KV TTL handles cleanup.

6TokenStorageService Façade

The façade maintains every existing gateway_id-based call-site signature unchanged. VaultTokenBackend receives the db session so it can resolve gateway_id → gateways.url internally via _resolve_mcp_url().

class TokenStorageService:
    def __init__(self, db: Session):
        self.db = db
        settings = get_settings()
        if settings.oauth_token_backend == "vault":
            self._backend = VaultTokenBackend(db, settings)   # db needed for _resolve_mcp_url
        elif settings.oauth_token_backend == "database":
            self._backend = DatabaseTokenBackend(db, settings)  # existing behaviour
        else:
            raise ValueError(
                f"Unknown OAUTH_TOKEN_BACKEND: {settings.oauth_token_backend}. "
                f"Expected 'database' or 'vault'."
            )

    # All public methods are one-line delegates — no logic here
    async def store_tokens(self, gateway_id, user_id, app_user_email,
                            access_token, refresh_token, expires_in, scopes):
        return await self._backend.store_tokens(
            gateway_id, user_id, app_user_email, access_token, refresh_token, expires_in, scopes)

    async def get_user_token(self, gateway_id, app_user_email, threshold_seconds=300):
        return await self._backend.get_user_token(gateway_id, app_user_email, threshold_seconds)

    async def get_token_info(self, gateway_id, app_user_email):
        return await self._backend.get_token_info(gateway_id, app_user_email)

    async def revoke_user_tokens(self, gateway_id, app_user_email):
        return await self._backend.revoke_user_tokens(gateway_id, app_user_email)

    async def cleanup_expired_tokens(self, max_age_days=30):
        return await self._backend.cleanup_expired_tokens(max_age_days)


# ── VaultTokenBackend: internal gateway_id → mcp_url resolution ─────────────
class VaultTokenBackend(AbstractTokenBackend):
    def __init__(self, db: Session, settings):
        self.db = db
        self.settings = settings
        # ... Vault client init (httpx or hvac) ...

    def _resolve_mcp_url(self, gateway_id: str) → str:
        """Resolve gateway_id → gateways.url. Used as Vault path key.
        Called once per operation — result is NOT cached (secrets may rotate)."""
        gateway = self.db.get(Gateway, gateway_id)
        if not gateway:
            raise ValueError(f"Gateway {gateway_id} not found")
        return gateway.url   # e.g. "https://mcp.github.acme.com"

    async def store_tokens(self, gateway_id: str, user_id: str, app_user_email: str, team_id: str, ...):
        mcp_url = self._resolve_mcp_url(gateway_id)  # ← resolution happens HERE, inside VaultTokenBackend
        server_id = self._hash_server_id(mcp_url)   # map mcp_url to stable path segment
        path = f"{self.mount}/data/{self.prefix}/{team_id}/{server_id}/{quote(app_user_email)}"
        # ... Vault KV v2 PUT ...

    async def get_user_token(self, gateway_id: str, app_user_email: str, team_id: str, ...):
        mcp_url = self._resolve_mcp_url(gateway_id)
        server_id = self._hash_server_id(mcp_url)
        path = f"{self.mount}/data/{self.prefix}/{team_id}/{server_id}/{quote(app_user_email)}"
        # ... Vault KV v2 GET on resolved path ...
Why this is correct:
  • Call sites unchanged — oauth_router.py, tool_service.py, gateway_service.py all pass gateway_id exactly as today.
  • Client never supplies gateway_id — the vault_router resolves server_id → server_tool_association → tools.gateway_id server-side and then calls TokenStorageService(db).store_tokens(gateway_id, …) — same as every other call site.
  • Vault path is human-readable — _resolve_mcp_url() converts the opaque UUID into the upstream URL before writing to Vault.
  • DatabaseTokenBackend unaffected — it uses gateway_id directly as a FK; no _resolve_mcp_url() needed.

7Vault Secret Schema

Path pattern:

<VAULT_KV_MOUNT>/data/<VAULT_KV_PATH_PREFIX>/<team_id>/<server_id>/<url-encoded email>

The path includes all three lookup keys: team_id (from user context), server_id (derived from mcp_url), and email (user identifier). This enables efficient lookup by any combination of these fields and provides natural isolation by team.

Path segment derivation:
  • team_id — extracted from authenticated user context (JWT claims or session data)
  • server_id — derived from gateway UUID or hashed from mcp_url for stable, human-friendly identifiers
  • email — URL-encoded user email from authentication context

Lookup implementation: Vault tokens are retrieved using the composite key (team_id, mcp_url, email). The service layer must:
  1. Extract team_id from authenticated user context (JWT claims or session)
  2. Convert mcp_url to server_id (hash or gateway UUID lookup)
  3. Construct Vault path: secret/contextforge/oauth/{team_id}/{server_id}/{email}
  4. Verify returned payload fields match the lookup keys

8New Client-Callable Vault Authorization Endpoints

These endpoints let a client trigger credential exchange using only the virtual server URL. They mirror the existing /oauth/authorize / /oauth/callback pair and are registered only when OAUTH_TOKEN_BACKEND=vault.

GET/vault/authorize/{server_id}

AspectDetail
Auth requiredYes — same ContextForge Bearer token used for MCP calls
Path paramserver_id — the ID from the client's virtual server URL
Optional query param?gateway_url=https://mcp.github.acme.com — selects one gateway when the virtual server has multiple OAuth gateways; omit for single-gateway servers
What it does 1. Looks up servers by server_id
2. Walks server_tool_association → tools.gateway_id → gateways
3. Reads gateway.oauth_config, builds PKCE + HMAC state, saves to oauth_states table
4. Redirects to IdP authorization URL
Response302 Redirect → IdP authorization URL

AspectDetail

5. Returns HTML success page

Gateway resolution helper

def _resolve_oauth_gateway(server: Server, db: Session,
                            preferred_url: str | None = None) → Gateway:
    rows = db.execute(
        select(Tool.gateway_id)
        .join(server_tool_association,
              server_tool_association.c.tool_id == Tool.id)
        .where(server_tool_association.c.server_id == server.id)
        .where(Tool.gateway_id.isnot(None))
        .distinct()
    ).scalars().all()                         # all gateway_ids on this server

    gateways = db.scalars(
        select(Gateway)
        .where(Gateway.id.in_(rows))
        .where(Gateway.auth_type == "oauth")
    ).all()

    if preferred_url:                         # ?gateway_url= supplied
        return next(g for g in gateways if g.url == preferred_url)
    return gateways[0]                        # first OAuth gateway

10Production — Vault with PostgreSQL Storage Backend

For production deployment, Vault uses PostgreSQL as its storage backend — secrets are persisted in a Postgres table (vault_kv_store). Vault encrypts all data before writing; Postgres stores only ciphertext. This provides high-availability, disaster recovery, and enterprise-grade persistence.
Note: The local development setup (docs/vault-local-dev-postgresql.md) also uses PostgreSQL backend for consistency with production. The main differences in production are: HA configuration, TLS encryption, auto-unseal with cloud KMS, and monitoring/alerting.

Auto-Unseal — Phase 2 Requirement

Without auto-unseal, every Vault restart requires manual input of 3/5 unseal key shards — causing downtime during incidents. Auto-unseal is required for production SLA > 99.9%.
# vault.hcl — AWS KMS auto-unseal
seal "awskms" {
  region     = "us-west-2"
  kms_key_id = "arn:aws:kms:us-west-2:123456789012:key/vault-unseal"

12Token Caching Strategy

To mitigate Vault API latency and reduce load on high-traffic deployments, ContextForge implements an optional in-memory token cache with configurable TTL. This cache is transparent to application code — TokenStorageService remains the single interface.

Cache Behavior

OperationCache StrategyRationale
get_token()Check cache → hit? return · miss? fetch from Vault + cacheRead-heavy workloads benefit most; tokens rarely change within TTL window
store_token()Write to Vault → invalidate cache entryEnsures fresh write is visible immediately on next read
refresh_token()Write to Vault → invalidate cache entryNew access token must be visible to ongoing calls
revoke_user_tokens()Delete from Vault → flush all matching cache entriesRevocation must propagate immediately

Configuration

# Enable token caching (default: false)
VAULT_TOKEN_CACHE_ENABLED=true

# Cache TTL in seconds (default: 300 = 5 minutes)
VAULT_TOKEN_CACHE_TTL=300

# Max cache entries (default: 10000)
VAULT_TOKEN_CACHE_MAX_SIZE=10000

Implementation Notes

  • Cache key: (team_id, server_id, email) — same composite key used for Vault lookups
  • Expiry: TTL-based eviction (default 5 min) + size-based LRU eviction at max capacity
  • Consistency: Write-through (write to Vault first, then cache) with immediate invalidation on update/delete
  • Thread safety: Cache implementation must be thread-safe for concurrent FastAPI workers
  • Metrics: Expose vault_token_cache_hit_total, vault_token_cache_miss_total, vault_token_cache_eviction_total
Trade-offs:
  • ✅ Pros: 10-50x read latency reduction (Vault ~10-20ms → in-memory ~0.5ms); reduces Vault API load; improves P99 latency for tool calls
  • ⚠️ Cons: Stale reads within TTL window if token rotated externally; increased memory footprint (~1KB per cached token); cache invalidation complexity
  • 📋 Recommendation: Enable for production deployments with >100 concurrent users or >500 tool calls/sec

Comparison with Database Backend

AspectDatabase BackendVault Backend (no cache)Vault Backend (with cache)
Read latency~2-5ms (connection pool)~10-20ms (HTTP + encryption)~0.5ms (memory hit) / ~10-20ms (miss)
CachingDB query cache (transparent)None (every read hits Vault)Application-layer TTL cache

Retry Logic (VaultTokenBackend)

async def get_user_token(self, gateway_id: str, app_user_email: str, team_id: str, ...):
    mcp_url = self._resolve_mcp_url(gateway_id)
    server_id = self._hash_server_id(mcp_url)
    path = f"{self.mount}/data/{self.prefix}/{team_id}/{server_id}/{quote(app_user_email)}"
    for attempt in range(3):
        try:
            return await self._vault_get(path)
        except httpx.ConnectTimeout:
            if attempt < 2:
                await asyncio.sleep(2 ** attempt)   # 1s, 2s
                continue
            logger.error("Vault unreachable after 3 attempts", extra={"gateway_id": gateway_id})
            raise GatewayConnectionError("Credential storage unavailable")
        except httpx.HTTPStatusError as e:
            if e.response.status_code == 403:
                logger.critical("Vault auth failure — VAULT_TOKEN invalid or expired")
                raise
            elif e.response.status_code == 404:
                return None   # No token stored — expected, caller redirects to /vault/authorize
            raise

§20.4 Operational Runbook — Rotate VAULT_TOKEN

When: Token TTL < 48 hours, or on suspected compromise. Expected duration: 15 min. Downtime: 0 (rolling restart).

export VAULT_ADDR=https://vault.acme.com:8200
export VAULT_TOKEN=<current-admin-token>  # pragma: allowlist secret

# 1. Create new scoped token
NEW_TOKEN=$(vault token create \
  -policy="contextforge" \
  -display-name="contextforge-prod-$(date +%Y%m%d)" \
  -ttl="720h" \
  -format=json | jq -r '.auth.client_token')

# 2a. Bare-metal: update .env on all nodes
sed -i.bak "s/VAULT_TOKEN=.*/VAULT_TOKEN=$NEW_TOKEN/" /etc/contextforge/.env

# 2b. Kubernetes: update secret
kubectl create secret generic contextforge-vault \
  --from-literal=token=$NEW_TOKEN \
  --dry-run=client -o yaml | kubectl apply -f -

# 3. Rolling restart (zero downtime)
kubectl rollout restart deployment/contextforge

# 4. Verify, then revoke old token
curl -H "Authorization: Bearer <cf-user-token>" https://contextforge.acme.com/health
vault token revoke <old-token>
