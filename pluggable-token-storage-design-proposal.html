Pluggable OAuth Token Storage — Design Proposal

ContextForge · Feature #5402 · Credential-Store Abstraction

1. Motivation & Product Owner Goal

Product Owner: "Break credentials storage into a module so we can plug-and-play different underlying storage classes (database, vault, etc.). Refactor database storage into independent table(s) if not already."

Today every OAuth access_token and refresh_token issued through the Authorization Code flow is hardwired into a single oauth_tokens DB table via TokenStorageService. The service directly imports SQLAlchemy models, runs queries, and drives encryption — all in one class with no seam to swap the storage layer. Clients who run HashiCorp Vault (or any future secret store) cannot use it without forking the core service.

6. AbstractTokenBackend Interface

class AbstractTokenBackend(ABC):

  # ── Write ──────────────────────────────────────────────────────────
  @abstractmethod
  async def store_tokens(
      gateway_id: str, user_id: str, app_user_email: str,
      access_token: str, refresh_token: Optional[str],
      expires_in: Optional[int], scopes: List[str],
  ) -> TokenRecord: ...

  # ── Read ───────────────────────────────────────────────────────────
  @abstractmethod
  async def get_user_token(
      gateway_id: str, app_user_email: str,
      threshold_seconds: int = 300,
  ) -> Optional[str]: ...

  # ── Metadata ───────────────────────────────────────────────────────
  @abstractmethod
  async def get_token_info(
      gateway_id: str, app_user_email: str,
  ) -> Optional[Dict[str, Any]]: ...

  # ── Revocation ─────────────────────────────────────────────────────
  @abstractmethod
  async def revoke_user_tokens(
      gateway_id: str, app_user_email: str,
  ) -> bool: ...

  # ── Maintenance ────────────────────────────────────────────────────
  @abstractmethod
  async def cleanup_expired_tokens(
      max_age_days: int = 30,
  ) -> int: ...

TokenRecord is a plain Python @dataclass (no SQLAlchemy dependency) returned by store_tokens. It carries: gateway_id, user_id, app_user_email, access_token, refresh_token, expires_at, scopes, token_type, created_at, updated_at.

7. TokenStorageService Façade — Constructor Change

class TokenStorageService:

  def __init__(self, db: Session):
    settings = get_settings()
    backend_name = settings.oauth_token_backend  # "database" | "vault"

    if backend_name == "vault":
      self._backend = VaultTokenBackend(settings)
    else:
      self._backend = DatabaseTokenBackend(db, settings)

  # All public methods are one-liner delegates:
  async def store_tokens(self, ...) -> TokenRecord:
    return await self._backend.store_tokens(...)

  async def get_user_token(self, ...) -> Optional[str]:
    return await self._backend.get_user_token(...)
  # ... and so on for all 5 methods

The db: Session argument is still accepted so all five call sites compile unchanged. When vault is selected the db is simply not forwarded to the backend — the Vault backend receives only the settings object.

9. VaultTokenBackend — Secret Schema

Each token pair is stored as a single Vault KV v2 secret at the path:

<VAULT_KV_MOUNT>/data/<VAULT_KV_PATH_PREFIX>/<gateway_id>/<url-encoded app_user_email>

Example: secret/data/contextforge/oauth/gw-github/alice%40acme.com

Secret payload (Vault data field):

KeyTypeNotes
access_tokenstringPlain-text. Vault encrypts at rest.
refresh_tokenstring | nullPlain-text. Omitted if not issued.
user_idstringOAuth provider user identifier.
token_typestringAlways "Bearer".
expires_atISO-8601 string | nullUTC expiry, or null.
scopesJSON array stringe.g. ["repo","read:user"].
created_atISO-8601 stringFirst issuance time.
updated_atISO-8601 stringLast write time.

No double-encryption. The DB backend encrypts because the oauth_tokens table stores ciphertext in a general-purpose relational DB. The Vault backend writes plain-text into Vault; Vault's own AES-256-GCM encryption provides at-rest protection, so adding a second Fernet layer would be redundant and would increase token size.

10. Token Refresh Strategy per Backend

BehaviourDatabaseTokenBackendVaultTokenBackend
Where refresh logic lives Extracted verbatim from today's _refresh_access_token() inside the backend class. Needs the DB session to read Gateway.oauth_config. Same refresh logic; reads Gateway.oauth_config via a separate DB lookup (Vault stores tokens only, not gateway config).
Updated token write-back UPDATE oauth_tokens SQL row. Vault KV PUT to the same path (creates a new version; KV v2 preserves history).
Cleanup SQL DELETE on expired rows (existing logic, unchanged). No-op + debug log. Vault lease TTLs or operator vault kv delete handle cleanup. Returns 0.

12. Dependency Strategy

No new mandatory runtime dependency.
The Vault backend uses httpx, which is already a runtime dependency (pyproject.toml:83).

An optional extra [vault] will add hvac>=2.3.0 for teams that prefer the official Python client. The backend detects hvac availability at import time; if absent it falls back to raw httpx calls to the Vault HTTP API. Either way the binary install is optional.
Install commandResult
pip install mcpgatewayDatabase backend only. Vault backend loads but will warn if enabled without Vault reachable.
pip install mcpgateway[vault]Adds hvac. Vault backend uses the official client.

13. Migration & Backward Compatibility

Default unchanged With OAUTH_TOKEN_BACKEND=database (the default), behaviour is identical to today. No operator action required.
Opt-in to Vault Operator sets OAUTH_TOKEN_BACKEND=vault plus VAULT_ADDR / VAULT_TOKEN. Gateway starts routing new tokens to Vault immediately.
Migrate existing DB tokens Existing rows in oauth_tokens are not automatically migrated. Users who already authorised will need to re-authorise once (OAuth callback will write the new token to Vault). A one-time migration script can be provided separately.
No Alembic migration needed The oauth_tokens table is untouched. If the operator later switches back to database, old DB rows are still present.

15. Open Questions for Product Owner

#QuestionImpact on design
Q1 Should switching from database → vault silently invalidate existing DB-stored tokens (forcing re-auth), or should we provide an automated migration script? If migration script required, add a CLI command in Phase 2.
Q2 Should the Vault backend also store the OAuth state (OAuthState table) in Vault, or keep that in DB? OAuth state is short-lived (10 min). Keeping it in DB is simpler; moving it to Vault is a separate concern.
Q3 Is AppRole auth (Phase 2) a blocker for the initial Vault backend, or is static token sufficient for the first release? Determines whether Phase 1 is usable in hardened production environments.
Q4 Should the cleanup_expired_tokens no-op for the Vault backend surface a warning, or be completely silent? Operators need to know that TTL management is now their responsibility when using Vault.
