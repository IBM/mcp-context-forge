Overall Assessment

This is a well-structured, security-conscious implementation. The façade pattern cleanly abstracts backend selection, the TOCTOU mitigation in vault_router.py is thoughtful, and the JWT-as-sole-authority for team scoping correctly preserves the Layer 1 token-scoping invariant from AGENTS.md. The Phase 1 design (DB backend accepts team_id but ignores it) is honest about its limitations and clearly documented.

Three blocking issues need to be resolved before merge, all in the Vault backend.

Blocking Issues

BLOCKING B1 — Stale cache entry returned after token refresh vault_backend.py:386–414

When a token is near-expiry and a refresh succeeds, _refresh_access_token() calls store_tokens() which invalidates the cache for the old entry. But get_user_token() returns the refreshed token immediately without writing it to the cache first. The next call will therefore hit Vault again (cache miss) rather than the in-memory copy, defeating the cache. Worse, if store_tokens() invalidates the key at one moment and then the current call stores nothing, a concurrent request can get a cache miss and issue a redundant Vault read during the window between invalidation and the next store.

Additionally, the LRU eviction logic on line 409–412 removes the insertion-order oldest key, not the least-recently-used key. Python dicts maintain insertion order (PEP 468 / CPython 3.7+), so frequently accessed entries inserted early are evicted before rarely accessed entries inserted later.

# Current (evicts oldest-inserted, not least-recently-used)
oldest_key = next(iter(VaultTokenBackend._token_cache))
VaultTokenBackend._token_cache.pop(oldest_key)

# Fix: use OrderedDict and move-to-end on access
from collections import OrderedDict
VaultTokenBackend._token_cache: OrderedDict = OrderedDict()

# On cache hit — move to end (most recent)
VaultTokenBackend._token_cache.move_to_end(cache_key)
return cached["token"]

# On cache write — add and evict first (least-recently-used)
VaultTokenBackend._token_cache[cache_key] = {...}
VaultTokenBackend._token_cache.move_to_end(cache_key)
if len(VaultTokenBackend._token_cache) > self.cache_max_size:
    VaultTokenBackend._token_cache.popitem(last=False)
BLOCKING B2 — Vault store does not preserve created_at on refresh vault_backend.py:282–303, _refresh_access_token:697–706

In store_tokens(), created_at is always set to datetime.now(timezone.utc). When _refresh_access_token() calls store_tokens() to persist the refreshed token, the original creation timestamp is silently overwritten. The DB backend preserves created_at by doing an UPSERT (updates only updated_at on existing rows). The Vault backend replaces the whole payload unconditionally.

This means token age cannot be audited via Vault, and any downstream logic that relies on created_at (e.g., max token age policies) will see a perpetually fresh timestamp. Fix: read the existing Vault record before writing, and preserve created_at if it already exists:

# In store_tokens():
existing = await self._vault_request("GET", path)
if existing and "data" in existing:
    original_created_at = existing["data"]["data"].get("created_at")
else:
    original_created_at = None

payload["data"]["created_at"] = original_created_at or now.isoformat()
BLOCKING B3 — Exception swallowing in vault_callback hides token storage failure vault_router.py:405–415, 440–457

Lines 405–415 call await token_storage.store_tokens(...) inside a try block whose except Exception on line 440 swallows all failures and returns a generic 500 HTML response. If Vault is unavailable or the token write fails, the OAuth callback appears to succeed (from the user's perspective) if the exception is caught after the store_tokens call but the HTML response on line 424 is returned. The current control flow is:

# Lines 347–415: all in the same try block
result = await no_storage_oauth_manager.complete_authorization_code_flow(...)
# ... state/context extraction ...
await token_storage.store_tokens(...)   # ← if this throws, falls through to ...
return HTMLResponse(content="✓ OAuth Authorization Successful", status_code=200)
# ... outer except returns 500 only if exception not caught before return

If store_tokens raises, execution jumps to the outer except Exception and the 500 response is returned — but the authorization code has already been consumed. The user will be shown an error and re-attempting auth will fail because the state is gone. This is unavoidable once the code is exchanged, but the success page should not be reachable without confirming storage succeeded. At minimum, check that store_tokens completed before returning the success page, and log a dedicated error distinguishing "exchange succeeded but storage failed" from "exchange failed".

Suggestions

SUGGESTION S1 — Vault 8-char server_id hash collision risk is real at scale vault_backend.py:113

_hash_server_id() truncates SHA-256 to the first 8 hex characters (32 bits). The birthday paradox means collisions become probable around sqrt(2^32) ≈ 65,536 distinct gateway URLs. For a large deployment with many gateways this is a real risk — two gateways with different URLs would silently share a Vault path, causing token cross-contamination. Consider using 16 characters (64-bit prefix) with negligible performance cost:

return hashlib.sha256(mcp_url.encode()).hexdigest()[:16]  # 2^64 space

Alternatively, URL-encode the full URL directly in the path. The SHA-256 approach is cleaner for Vault path length limits but the truncation depth should be justified in a comment.

SUGGESTION S2 — build_token_user_context collapses empty list to None, hiding a distinct state token_storage_service.py:86

Line 86: effective_teams = token_teams if token_teams else None collapses both None (missing teams key, Admin UI) and [] (explicit public-only token) to None, which maps both to the shared Vault path. The docstring acknowledges this, but the token-scoping invariant table in AGENTS.md treats them differently: None = Admin bypass, [] = public-only. A public-only token stored in the shared Admin path could be retrieved by an admin traversing the Vault hierarchy. If these two cases are intentionally collapsed for Vault storage (reasonable for Phase 1), add a comment that justifies it explicitly; otherwise distinguish the paths (e.g., shared-admin vs. shared-public).

SUGGESTION S3 — teams[0] authority decision lacks a test for ordering regression token_storage_service.py:183–191

The critical correctness assumption is that AuthMiddleware always returns teams in the same order (sorted by EmailTeamMember.id ascending). The warning log at line 183–191 is a good signal, but it fires at WARNING level and only when the Vault backend is active and the user has more than one team. There are no tests that verify ordering consistency on the middleware side — a refactor of the team query (e.g., adding ORDER BY team_name for display purposes) could silently break token lookup for multi-team users. Worth adding a test fixture with a multi-team user to assert _get_team_id() returns the same team ID across multiple requests with the same JWT.

SUGGESTION S4 — _vault_request silently returns None on 404 and also on empty success body vault_backend.py:168–249

The method returns None in two distinct cases: HTTP 404 (token not found) and an empty/non-JSON body from Vault. Callers like get_user_token() treat both as "no token", which is correct for 404 but masks a Vault misconfiguration (wrong mount, wrong path prefix, Vault returning HTML error pages) for the second case. Consider distinguishing the two, or at least logging at WARNING when a non-404 non-success response results in a None return:

if response.status_code == 404:
    return None
if not response.content:
    logger.warning("Vault returned empty body for %s %s (status=%d)", method, path, response.status_code)
    return None
SUGGESTION S5 — RFC 8707 resource normalization applied inconsistently between backends vault_backend.py:669–678, db_backend.py:_refresh_access_token

The Vault backend's _refresh_access_token() calls normalize_resource_url() from base.py (shared utility). The DB backend has the same normalization logic but it was the source of the shared function, so verify both are calling the shared version (no stale inline copy remains). If there are any callers in the DB backend that still define _normalize_resource_url inline, they should be replaced with the shared import.

Minor Notes

NOTE N1 — vault_router.py re-implements RFC 8707 URL normalization inline vault_router.py:338–344

Lines 338–344 import urlparse/urlunparse and strip the fragment/query inline rather than calling normalize_resource_url() from base.py. Consolidate to one call site.

NOTE N2 — _resolve_mcp_url duplicated in both backends db_backend.py:84–99, vault_backend.py:84–102

Both backends define an identical _resolve_mcp_url(gateway_id) method. Could be a static method on AbstractTokenBackend or a standalone helper, avoiding drift if the Gateway model changes.

NOTE N3 — cleanup_warned sentinel prevents seeing the warning on backend restart vault_backend.py:517–520

The class-level _cleanup_warned flag means the warning fires once per process lifetime. In a container/Kubernetes environment where pods restart frequently, this is fine. In a long-running process the intent is log deduplication, which is correct. Consider INFO level instead of WARNING since a no-op cleanup is expected and intentional for the Vault backend — WARNING suggests something is wrong.

NOTE N4 — Import-outside-toplevel pattern in vault_callback vault_router.py:395

from mcpgateway.db import EmailUser is imported inside the function body with an inline pylint disable comment. This is the established pattern in the project to avoid circular imports at module load time, and is acceptable. Worth noting for consistency — other functions in the same module do not need this import and it could be hoisted to the top of the file once circular import constraints are resolved.
