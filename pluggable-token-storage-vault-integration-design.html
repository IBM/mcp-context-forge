Pluggable Token Storage — Vault Integration Design

ContextForge · Feature #5402 · Grounded in: Alice (GitHub) + Bob (Jira) Multi-User Virtual Server Scenario

1. Business Problem (from scenario)

A client runs ContextForge to host all their MCP servers. They register GitHub MCP and Jira MCP as OAuth gateways, then expose them together as a single virtual server vs-engineering. Multiple users — Alice and Bob — authenticate against different gateways and invoke tools through that virtual server.

The client's requirement: They operate HashiCorp Vault as their enterprise secret store. They do not want OAuth tokens (Alice's GitHub token, Bob's Jira token) to ever land in the ContextForge application database. All credentials must live in Vault, managed under their existing Vault policies, audited via Vault audit logs, and rotated via Vault leases.

Today, store_tokens() always writes to the oauth_tokens DB table. There is no seam to redirect that write to Vault. The design below introduces that seam.

4. TokenStorageService as a Thin Façade

The constructor is the only thing that changes. All five call sites continue to call TokenStorageService(db) identically — the db argument is accepted for backward compatibility and forwarded to the DB backend only when the database backend is active.

class TokenStorageService:
    def __init__(self, db: Session):
        settings = get_settings()
        backend  = settings.oauth_token_backend   # "database" (default) | "vault"

        if backend == "vault":
            self._backend = VaultTokenBackend(settings)   # db ignored for Vault
        else:
            self._backend = DatabaseTokenBackend(db, settings)  # existing behaviour

    # Every public method is a one-line delegate — no logic here
    async def store_tokens(self, ...)   → self._backend.store_tokens(...)
    async def get_user_token(self, ...) → self._backend.get_user_token(...)
    async def get_token_info(self, ...) → self._backend.get_token_info(...)
    async def revoke_user_tokens(self, ...) → self._backend.revoke_user_tokens(...)
    async def cleanup_expired_tokens(self, ...) → self._backend.cleanup_expired_tokens(...)

5. AbstractTokenBackend — The Contract Both Backends Satisfy

class AbstractTokenBackend(ABC):

    # Write — called at OAuth callback after IdP returns code + tokens
    @abstractmethod
    async def store_tokens(gateway_id, user_id, app_user_email,
                           access_token, refresh_token, expires_in, scopes) → TokenRecord

    # Read — called on every tool call / health-check / resource fetch
    @abstractmethod
    async def get_user_token(gateway_id, app_user_email,
                             threshold_seconds=300) → Optional[str]

    # Metadata — called by admin status API
    @abstractmethod
    async def get_token_info(gateway_id, app_user_email) → Optional[Dict]

    # Revocation — called at user logout or admin revoke
    @abstractmethod
    async def revoke_user_tokens(gateway_id, app_user_email) → bool

    # Maintenance — called by scheduled cleanup job
    @abstractmethod
    async def cleanup_expired_tokens(max_age_days=30) → int

TokenRecord is a plain @dataclass (no SQLAlchemy). It carries gateway_id, user_id, app_user_email, access_token, refresh_token, expires_at, scopes, token_type, created_at, updated_at. Returned by store_tokens() so callers never touch a SQLAlchemy ORM object directly.

6. Vault Secret Schema — Where Alice's and Bob's Tokens Live

Vault secret path pattern:

<VAULT_KV_MOUNT>/data/<VAULT_KV_PATH_PREFIX>/<gateway_id>/<url-encoded email>

One secret per (gateway_id, user_email) — identical isolation model to the DB's UNIQUE (gateway_id, app_user_email) constraint.
UserGatewayVault secret path
alice@acme.com gw-github secret/data/contextforge/oauth/gw-github/alice%40acme.com
bob@acme.com gw-jira secret/data/contextforge/oauth/gw-jira/bob%40acme.com

Vault secret payload (the data field inside the KV v2 secret):

{
  "access_token":  "gho_Aa1Bb2...",        // plain-text — Vault encrypts at rest
  "refresh_token": "ghr_Rr9Ss8...",        // plain-text, nullable
  "user_id":       "github_uid_44712",     // OAuth provider identifier
  "token_type":    "Bearer",
  "expires_at":    "2025-06-10T18:00:00Z", // ISO-8601 UTC, or null
  "scopes":        '["repo","read:user"]', // JSON-encoded list
  "created_at":    "2025-06-10T10:00:00Z",
  "updated_at":    "2025-06-10T10:00:00Z"
}

7. Scenario Walkthrough — Every Step with Vault Backend Active

The operator has set OAUTH_TOKEN_BACKEND=vault. All other steps (gateway registration, virtual server creation, PKCE flow) are identical.

Step 1 — Admin registers gw-github and gw-jira

No change. OAuth config (client_id, client_secret, URLs) stays in the gateways DB table, encrypted by EncryptionService as before. Vault stores user tokens only — not gateway config.

Step 2 — Alice authenticates with GitHub (OAuth callback writes to Vault)

Alice → GET /oauth/authorize/gw-github
Unchanged. ContextForge builds PKCE + HMAC state, saves state to oauth_states table (short-lived, stays in DB), redirects to GitHub.
GitHub redirects back → GET /oauth/callback?code=abc&state=…
Unchanged. ContextForge validates HMAC state, exchanges code for tokens with GitHub IdP. Receives { access_token, refresh_token, expires_in: 28800 }.
VAULT store_tokens() writes to Vault instead of DB
VaultTokenBackend.store_tokens("gw-github", "github_uid_44712", "alice@acme.com", "gho_Aa1Bb2...", "ghr_Rr9Ss8...", 28800, ["repo","read:user"])

Internally: PUT secret/data/contextforge/oauth/gw-github/alice%40acme.com with payload above.
No application-layer encryption — tokens are plain-text to Vault; Vault's AES-256-GCM seal protects them at rest.
No row is written to oauth_tokens DB table.
VAULT "Fetch Tools" triggers get_user_token() from Vault
GET secret/data/contextforge/oauth/gw-github/alice%40acme.com
Checks expires_at from the Vault payload — token is fresh → returns "gho_Aa1Bb2..." plain-text.
Gateway service uses it as Authorization: Bearer gho_Aa1Bb2... to sync tool list from GitHub MCP.

Step 3 — Bob authenticates with Jira

VAULT store_tokens() writes Bob's Jira token to Vault
PUT secret/data/contextforge/oauth/gw-jira/bob%40acme.com

Vault now contains two isolated secrets: 
secret/data/contextforge/oauth/gw-github/alice%40acme.com  → Alice's GitHub token
secret/data/contextforge/oauth/gw-jira/bob%40acme.com      → Bob's Jira token

Step 4 — Alice calls github__search_code via vs-engineering

tool_service resolves tool → belongs to gw-github, grant_type = authorization_code
Code path unchanged: TokenStorageService(token_db).get_user_token("gw-github", "alice@acme.com")
VAULT VaultTokenBackend fetches from Vault
GET secret/data/contextforge/oauth/gw-github/alice%40acme.com
Reads expires_at → not expired → returns "gho_Aa1Bb2..."
No DB query. No decryption step (token was stored plain-text).
ContextForge forwards request with Alice's token
POST https://mcp.github.acme.com/sse with Authorization: Bearer gho_Aa1Bb2.... GitHub sees Alice's identity.

Step 5 — Bob calls jira__create_issue via vs-engineering

VAULT VaultTokenBackend fetches Bob's Jira token
GET secret/data/contextforge/oauth/gw-jira/bob%40acme.com
Returns Bob's Jira access token. Alice's path is never queried — strict path-per-user isolation.
ContextForge forwards with Bob's token
POST https://mcp.jira.acme.com/… with Bob's Bearer token. Jira creates the issue under Bob's account.

Step 6 — Transparent Token Refresh (Alice's token nears expiry)

VAULT get_user_token() detects near-expiry from Vault metadata
Reads expires_at from Vault payload.
now + 300s >= expires_at → True → triggers refresh.
Refresh call to GitHub IdP
VaultTokenBackend reads gateway.oauth_config from DB (gateway config stays in DB — only user tokens move to Vault).
Calls OAuthManager.refresh_token(refresh_token, oauth_config) → GitHub issues new { access_token, expires_in }.
VAULT Write-back: updated tokens saved as a new Vault KV version
PUT secret/data/contextforge/oauth/gw-github/alice%40acme.com with new access_token and updated expires_at.
Vault KV v2 automatically keeps the previous version — full audit history retained without any extra work.

Step 7 — Cross-User Isolation (Alice tries Jira without authorising)

VAULT GET secret/data/contextforge/oauth/gw-jira/alice%40acme.com → 404
Path does not exist in Vault. VaultTokenBackend returns None.
ContextForge raises: "No OAuth tokens found for user alice@acme.com on gateway Jira MCP. Visit /oauth/authorize/gw-jira".
Alice cannot access Bob's path — Vault policies can enforce this at the Vault layer too (see §9).

Step 8 — Revocation (Bob logs out)

VAULT revoke_user_tokens() deletes the Vault secret
DELETE secret/data/contextforge/oauth/gw-jira/bob%40acme.com
(Vault KV v2 soft-delete; hard-delete available via metadata endpoint.)
Alice's GitHub path is untouched.

Step 9 — Cleanup (scheduled job)

cleanup_expired_tokens() is a no-op for the Vault backend.
Vault does not support "list all secrets older than N days" without iterating the entire tree — that is expensive and not idiomatic. Instead, operators configure Vault KV lease TTLs to auto-expire stale secrets, or use vault kv delete in a Vault agent policy. The method returns 0 and logs a DEBUG message. The existing DB-backend cleanup job continues working unchanged for teams on the database backend.

9. Vault Policy — What ContextForge Needs

The ContextForge application token needs exactly these capabilities on the KV mount:

path "secret/data/contextforge/oauth/*" {
  capabilities = ["create", "update", "read", "delete"]
}

path "secret/metadata/contextforge/oauth/*" {
  capabilities = ["delete", "list"]   // needed for hard-delete and future list
}

No wildcard admin access. ContextForge only touches secrets under its own prefix. Gateway config, other application secrets, and infra secrets are unreachable with this policy.

11. Migration Path for Existing Deployments

Keep Database Backend
  • Do nothing. Default is database.
  • All existing Alice/Bob tokens continue to work.
  • No Alembic migration. No config change.
Switch to Vault Backend
  • Set OAUTH_TOKEN_BACKEND=vault + Vault env vars.
  • Existing DB tokens are not auto-migrated.
  • Users re-authorise once — new tokens go to Vault.
  • Old oauth_tokens rows can be manually purged after migration.
No dual-backend mode in Phase 1. When Vault is active, the DB backend is not consulted as a fallback. The operator must plan the migration window. A one-time migration script (reads DB rows, writes to Vault, deletes DB rows) is a Phase 2 deliverable.
