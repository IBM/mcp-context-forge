Pluggable Token Storage — Vault Integration Design

ContextForge · Feature #5402 · Revised: MCP URL as Vault key · Client-callable /vault/authorize · server_tool_association resolution

1. What Changed from the Original Design

Three revisions from the original draft:

  1. Vault path key = MCP URL, not gateway_id. Clients connect exclusively via virtual server URL (http://host:4444/servers/{server_id}/mcp). They never know a gateway_id. The Vault secret path now uses the upstream gateway URL resolved from the DB — which is the actual MCP endpoint the credential is for — as a human-readable, stable identifier.
  2. Resolution chain is explicit. server_id → server_tool_association → tool.gateway_id → gateways.url. This is how ContextForge finds the real upstream MCP URL without the client ever exposing or knowing a gateway ID.
  3. New client-callable Vault auth endpoints. GET /vault/authorize/{server_id} and GET /vault/callback mirror the existing /oauth/authorize / /oauth/callback pair, but are scoped to the virtual server URL the client already has.

2. Client Access Model — Why gateway_id Is Never Exposed

A client's MCP configuration looks like this — it knows only the virtual server URL:

# Client MCP config (claude_desktop_config.json, cursor settings, etc.)
{
  "servers": {
    "git-server": {
      "type": "streamable-http",
      "url": "http://localhost:4444/servers/647ad7b348044bce8fa27a2157b00a0d/mcp",
      "headers": { "Authorization": "Bearer your-token-here" }
    }
  }
}

The server_id (647ad7b3…) is the virtual server's primary key in the servers table. The underlying gateway_id values are internal — they never appear in client config and clients must not need to supply them.

For a virtual server with tools from multiple gateways (e.g. gw-github and gw-jira), this resolution finds all distinct gateway URLs. Each gets its own Vault secret path. The client only ever presents the single virtual server URL and their ContextForge Bearer token.

3. Vault Secret Schema — MCP URL as the Path Key

Vault secret path pattern (revised):

<VAULT_KV_MOUNT>/data/<VAULT_KV_PATH_PREFIX>/<url-encoded upstream MCP URL>/<url-encoded email>

The first path segment is the upstream gateway URL fetched from gateways.url via the resolution chain above — not the gateway_id UUID, not the virtual server URL. This makes secrets human-readable and stable across gateway ID changes.
UserVirtual Server (client sees)Resolved upstream MCP URLVault secret path
alice@acme.com /servers/647ad7b3…/mcp https://mcp.github.acme.com secret/data/contextforge/oauth/https%3A%2F%2Fmcp.github.acme.com/alice%40acme.com
bob@acme.com /servers/647ad7b3…/mcp https://mcp.jira.acme.com secret/data/contextforge/oauth/https%3A%2F%2Fmcp.jira.acme.com/bob%40acme.com

Vault secret payload — mirrors the oauth_tokens DB table columns exactly, with mcp_url replacing gateway_id as the system identifier (matching the System column visible in the token management UI):

{
  "mcp_url":       "https://mcp.github.acme.com",   // replaces gateway_id — the upstream MCP URL
  "access_token":  "gho_Aa1Bb2...",                  // plain-text — Vault encrypts at rest
  "refresh_token": "ghr_Rr9Ss8...",                  // plain-text, nullable
  "user_id":       "github_uid_44712",              // OAuth provider identifier
  "token_type":    "Bearer",
  "expires_at":    "2025-06-10T18:00:00Z",          // ISO-8601 UTC, or null
  "scopes":        '["repo","read:user"]',          // JSON-encoded list
  "created_at":    "2025-06-10T10:00:00Z",
  "updated_at":    "2025-06-10T10:00:00Z"
}
Why mcp_url and not gateway_id? The gateway_id is an internal UUID opaque to clients and operators. The upstream MCP URL (gateways.url) is what the secret actually protects access to — it maps directly to what the UI's System column should display, and it is stable and meaningful for audit purposes (e.g. "alice's token for https://mcp.github.acme.com").

4. New Client-Callable Vault Authorization Endpoints

These two endpoints let a client trigger the full Vault credential exchange using only the virtual server URL they already have — no gateway knowledge required. They mirror the existing /oauth/authorize / /oauth/callback pair.

New router: mcpgateway/routers/vault_router.py — prefix /vault, tag vault
Registered in main.py alongside oauth_router. Enabled only when OAUTH_TOKEN_BACKEND=vault.

Endpoint 1 — Initiate Vault Auth Flow

GET/vault/authorize/{server_id}

AspectDetail
Auth requiredYes — same ContextForge Bearer token the client already uses for MCP calls
Path paramserver_id — the ID from the client's virtual server URL (/servers/{server_id}/mcp)
What it does 1. Looks up servers table by server_id.
2. Walks server_tool_association → tools.gateway_id → gateways to find all OAuth gateways.
3. If multiple OAuth gateways, accepts optional ?gateway_url=… to select one; otherwise selects the first.
4. Reads gateway.oauth_config, builds PKCE + HMAC state (same as /oauth/authorize).
5. Stores state in oauth_states table with the resolved gateway_id.
6. Redirects to the IdP authorization URL.
Response302 Redirect → IdP authorization URL (same as /oauth/authorize/{gateway_id})
Errors404 server not found · 400 no OAuth-enabled gateways on this server · 403 caller lacks access to server
# Pseudocode — vault_router.py
@vault_router.get("/authorize/{server_id}")
async def initiate_vault_auth(server_id: str, request: Request,
                               gateway_url: str | None = None,
                               current_user = Depends(get_current_user_with_permissions),
                               db: Session = Depends(get_db)):

    # 1. Resolve server → gateways via server_tool_association
    server = db.get(Server, server_id)              # 404 if missing
    gateway = _resolve_oauth_gateway(server, db, preferred_url=gateway_url)

    # 2. Reuse existing OAuth flow initiation (no duplication)
    oauth_manager = OAuthManager(token_storage=TokenStorageService(db))
    auth_data = await oauth_manager.initiate_authorization_code_flow(
        gateway.id, gateway.oauth_config,
        app_user_email=requester_email
    )
    return RedirectResponse(url=auth_data["authorization_url"])

Endpoint 2 — Vault Auth Callback

GET/vault/callback

AspectDetail
Auth requiredNo — IdP redirects here with code + state; state carries the session
Query paramscode, state, error, error_description — identical to /oauth/callback
What it does 1. Validates HMAC state, resolves gateway_id from oauth_states table (identical to /oauth/callback).
2. Exchanges code for tokens via IdP.
3. Resolves mcp_url = gateways.url from the gateway record.
4. Calls VaultTokenBackend.store_tokens(mcp_url, user_id, email, access_token, …) — writes to Vault path …/<encoded mcp_url>/<encoded email>.

Key difference from /oauth/callbackStep 4 always writes to Vault regardless of OAUTH_TOKEN_BACKEND setting — this endpoint is only registered when OAUTH_TOKEN_BACKEND=vault.

Helper: _resolve_oauth_gateway

def _resolve_oauth_gateway(server: Server, db: Session,
                            preferred_url: str | None = None) -> Gateway:
    """Walk server_tool_association → tools.gateway_id → gateways.
    Returns the first active OAuth gateway on this virtual server."""

    # Pull distinct gateway_ids from tools linked to this server
    rows = db.execute(
        select(Tool.gateway_id)
        .join(server_tool_association,
              server_tool_association.c.tool_id == Tool.id)
        .where(server_tool_association.c.server_id == server.id)
        .where(Tool.gateway_id.isnot(None))
        .distinct()
    ).scalars().all()

    if not rows:
        raise HTTPException(400, "No gateways associated with this server")

    gateways = db.scalars(select(Gateway)
        .where(Gateway.id.in_(rows))
        .where(Gateway.auth_type == "oauth")).all()

    if not gateways:
        raise HTTPException(400, "No OAuth-enabled gateways on this server")

    if preferred_url:
        match = next((g for g in gateways if g.url == preferred_url), None)
        if not match:
            raise HTTPException(400, "Requested gateway_url not found on this server")
        return match

    return gateways[0]   # first OAuth gateway wins

5. End-to-End Flow: Alice Authorizes via /vault/authorize

Alice's MCP client knows only: http://localhost:4444/servers/647ad7b3…/mcp

Alice → GET /vault/authorize/647ad7b3…
ContextForge resolves: server_id=647ad7b3… → server_tool_association → tools.gateway_id=gw-github → gateways.url=https://mcp.github.acme.com.
Reads gw-github.oauth_config, builds PKCE + HMAC state, saves to oauth_states table (stays in DB — short-lived).
302 Redirect → GitHub IdP authorization URL
Alice is redirected to GitHub to approve access. Identical UX to existing /oauth/authorize flow.
GitHub → GET /vault/callback?code=abc&state=…
ContextForge validates HMAC state. Resolves gateway_id from oauth_states table. Fetches gateway record → gateways.url = https://mcp.github.acme.com. Exchanges code for tokens with GitHub IdP.
VAULT VaultTokenBackend.store_tokens(mcp_url, …)
mcp_url = "https://mcp.github.acme.com" (from gateways.url)
Vault path: PUT secret/data/contextforge/oauth/https%3A%2F%2Fmcp.github.acme.com/alice%40acme.com
Payload stores mcp_url (what the System column shows in the UI), access_token, refresh_token, expires_at, scopes, etc.
No row written to oauth_tokens DB table. Vault's AES-256-GCM seal protects at rest.
Success page returned to Alice's browser
HTML success page with link back. Alice's MCP client can now call tools through /servers/647ad7b3…/mcp — ContextForge will fetch her token from Vault on each tool call.

When ?gateway_url is omitted and the server has only one OAuth gateway, that gateway is chosen automatically — no extra query param needed for the simple single-gateway case.

10. AbstractTokenBackend — Revised Signature

The interface now takes mcp_url instead of gateway_id as the resource discriminator. The DatabaseTokenBackend resolves mcp_url → gateway_id internally via a DB lookup so the existing oauth_tokens table is unchanged.

class AbstractTokenBackend(ABC):

    @abstractmethod
    async def store_tokens(
        mcp_url: str,          # gateways.url — replaces gateway_id in the public API
        user_id: str,          # OAuth provider user ID
        app_user_email: str,   # ContextForge user email
        access_token: str,
        refresh_token: str | None,
        expires_in: int | None,
        scopes: list[str]
    ) → TokenRecord

    @abstractmethod
    async def get_user_token(
        mcp_url: str,
        app_user_email: str,
        threshold_seconds: int = 300
    ) → str | None

    @abstractmethod
    async def get_token_info(mcp_url: str, app_user_email: str) → dict | None

    @abstractmethod
    async def revoke_user_tokens(mcp_url: str, app_user_email: str) → bool

    @abstractmethod
    async def cleanup_expired_tokens(max_age_days: int = 30) → int
DatabaseTokenBackend compatibility note. The existing call sites pass gateway_id (a UUID). The DB backend resolves mcp_url → gateway_id with a single SELECT id FROM gateways WHERE url = :mcp_url lookup before each operation. This is a single extra DB round-trip per token operation — acceptable given token ops are infrequent relative to tool calls.
